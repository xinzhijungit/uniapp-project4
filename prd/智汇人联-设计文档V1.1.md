二、产品功能设计
2.1 功能一：人联研判功能
2.1.1 背景
当前公安业务系统中积累了海量的警情、人员、车辆数据，但数据价值未被充分挖掘，尤其是在风险研判方面，依赖人工经验进行关联分析，效率低且容易遗漏线索。需建设一个基于知识图谱的AI风险研判工具，实现对高危人员的智能识别、多维画像和关联分析。
2.1.2 目标
构建一个集人员/车辆/警情多维检索、AI风险智能研判、知识图谱可视化分析于一体的功能模块。用户输入身份证、车牌、警情编号或手机号后，系统能：
1.  自动检索并展示目标对象的全面信息、风险标签、画像及处置建议（如”原型图：人联研判1.png“左侧）。
2.  动态生成以目标对象为中心的知识图谱，直观展示其与民警、警情、地点、其他人员等实体的复杂关系（如”原型图：人联研判2.png“）。

（2） 图谱构建流程
当用户发起检索（例如，输入身份证号）时，后端执行以下逻辑：
1.  定位核心节点
：根据输入的身份证号，在 FKD\_BJR 表中找到对应的市民节点 N\_citizen。
1.  1跳关系扩展
：
*   查询
 JQ\_SMSJ\_ontology 表，找到所有 BJR\_BH 等于 N\_citizen.id 的记录，获取关联的 JJDBH 列表（警情节点 N\_case1...N\_caseN）。
*   查询
 JQ\_MJFZ\_ontology 表，找到所有 JJDBH 在上一步警情节点列表中的记录，获取负责这些警情的 MJGH 列表（民警节点 N\_police1...N\_policeN）。
1.  2跳关系及属性扩展
：
*   对于每一个查询到的警情节点
 N\_case，查询 JQ\_ZTLZ\_ontology 获取其当前状态，作为节点属性。
*   对于每一个查询到的民警节点
 N\_police，查询 MJ\_XX\_ontology 获取其职能和状态，作为节点属性。
*   （可选）分析
 JQ\_SMSJ\_ontology，找出与 N\_citizen 涉及同一警情的其他市民节点，构建“同案人员”关系。
2.  AI风险解析与关系补充
：
*   读取
 N\_citizen 的 AI\_TAG\_ontology 字段。
*   使用正则表达式或NLP规则解析标签内容。例如，匹配“与前科人员同行3次”，则创建或更新与虚拟/实有“前科人员”节点的关系。
*   匹配“网购行窃工具”，则创建“购买”关系节点。
3.  生成图谱数据
：将上述查询到的所有节点（市民、警情、民警、地点、物品）和关系（涉及、负责、同行、购买）组装成图数据结构（如 {nodes: \[...], links: \[...]}），返回给前端渲染。

4.  路径长度为3的关联查询
：支持查询中心节点到目标节点路径长度为3的所有关联关系。例如：
   - 李明 --户主--> H001（户籍）
   - 王丽 --配偶--> H001（户籍）
   - 王丽 --有案底--> 案底记录
：通过户籍关系关联查询同户籍人员的案底记录，实现更全面的人员关联分析。
：应用场景包括智能标签测试/批量测试中的"图谱数据"列表展示，以及人联研判的"西安警务本体关联知识可视化"模块。

2.2 功能二：智能标签功能
2.2.1 背景
公安业务中产生大量警情、人员、民警数据，但缺乏动态、精细化的标签体系进行风险分级和画像刻画。传统手工标注效率低，且规则僵化。引入大语言模型，结合灵活配置的提示词，可以实现对报警人、警情、民警三类核心实体的自动化、智能化标签标注，支撑风险预警、警力调度和案件研判。
2.2.2 目标
建设一个智能标签配置管理平台，业务人员可以：
1.  动态创建/编辑标签
：定义标签的名称、标识符、优先级、适用对象（报警人/警情/民警）及分类。
2.  维护AI提示词
：为每个标签编写自然语言的判定规则提示词，并设置正例/反例用于测试。
3.  批量测试与全量标注
：支持对单条或批量数据进行测试，并通过定时任务或触发机制，调用AI模型为实体自动添加标签。
4.  标签生命周期管理
：支持启用、禁用、编辑、删除标签，标签变更后能重新触发标注任务。
最终实现标签数据落库到业务表的 AI\_TAG\_ontology 字段（JSON格式，存储多个标签编码），并支持前端按标签检索、统计和风险预警。

（1)数据准备

(2) 标注执行逻辑
针对一个目标实体（如某报警人）和一组已启用的标签（仅限该实体类型的标签），系统执行以下步骤：
1.  筛选标签
：根据标签的适用对象（报警人/警情/民警）和启用状态，获取需要判断的标签列表。
2.  构造Prompt
：对每个标签，将其提示词中的 {{entity}} 替换为该实体的文本描述。
3.  调用AI模型
：将构造好的Prompt发送给DeepSeek（或其他配置的模型），要求模型返回结构化结果（例如 是 或 否，或直接返回标签代码列表）。
4.  结果聚合
：收集所有标签的判断结果，将命中的标签标识符（如 \["HIGH\_RISK\_PERSON", "NOCTURNAL"]）存入对应业务表的 AI\_TAG\_ontology 字段，格式为JSON数组。
5.  落库
：更新 FKD\_BJR.AI\_TAG\_ontology、JJD\_JJD.AI\_TAG\_ontology、MJ\_XX\_ontology.AI\_TAG\_ontology。

(3)触发方式
*   手动触发
：在标签编辑页点击【批量测试】或针对单条实体的【立即标注】。
*   自动定时任务
：每日凌晨执行，对最近24小时新增或更新的实体进行增量标注。
*   事件触发
：当警情办结、人员信息变更、新反馈单提交时，触发相关联实体的重新标注（异步）。

(4)标签数据使用
*   前端展示
：在人员/警情/民警详情页，展示其标签列表，并按优先级高亮高危标签。
*   风险预警
：当实体命中高优先级标签（如“高危人员”、“扬言报复社会人员”）时，自动推送到盯办列表或指挥大屏。
*   统计分析
：支持按标签维度统计数量、趋势，辅助决策。

(5)数据模型设计
a. 标签配置表 TAG\_CONFIG\_ontology用于存储所有可配置的智能标签。
b.标签测试示例表 TAG\_EXAMPLE\_ontology存储每个标签的正例/反例，用于测试准确率。

*   JJD\_JJD.AI\_TAG\_ontology json  存储警情标签。
*   MJ\_XX\_ontology.AI\_TAG\_ontology json 存储民警标签。

**【重要】AI\_TAG\_ontology 标签累计规则：**
*   **存储格式**：JSON数组，如 `["_gzry", "_glzt"]`
*   **累计逻辑**：
    *   同一个报警人/警情可能匹配多个智能标签
    *   每次测试或批量测试时，先读取现有的 AI\_TAG\_ontology 字段值
    *   将新匹配的标签追加到现有标签数组中（避免重复）
    *   最后将合并后的标签数组更新回数据库
*   **MySQL JSON字段特性**：
    *   MySQL 会自动将 JSON 类型字段解析成 JavaScript 对象/数组
    *   读取时需判断数据类型：如果是数组则直接使用，如果是字符串则需要 JSON.parse

七、业务规则文档
7.1 风险评分算法规则
7.1.1 评分计算公式
风险总分 = Σ(标签权重 × 标签得分) + 前科记录得分 + 涉警频率得分 + 异常行为得分 + 社交网络风险得分
7.1.2 标签得分计算
| 标签优先级 | 权重 | 得分范围 |
| :--- | :--- | :--- |
| 高 | 0.4 | 80-100 |
| 中 | 0.3 | 50-79 |
| 低 | 0.2 | 1-49 |
7.1.3 前科记录得分
前科记录得分 = 前科类型系数 × 前科次数 × 时间衰减系数
| 前科类型 | 系数 |
| :--- | :--- |
| 严重暴力犯罪 | 1.0 |
| 财产犯罪 | 0.7 |
| 治安违法 | 0.3 |
时间衰减系数 = 1 - (当前时间 - 前科时间) / 3650 (最多衰减至0.3)

7.1.4 涉警频率得分
涉警频率得分 = 近30天涉警次数 × 10 (最高50分)
7.1.5 异常行为得分
异常行为包括：昼伏夜出、频繁更换住址、使用虚假身份等

7.2 标签判定规则
7.2.1 高危人员判定规则
- 有严重暴力犯罪前科
- 涉毒人员
- 扬言报复社会
- 被列为重点人员且有极端倾向
- 多次购买可疑物品（如开锁工具、管制刀具）

7.2.2 重点人员判定规则
- 有犯罪前科记录
- 多次参与治安案件
- 被公安机关列为关注对象

7.2.3 关注人员判定规则
- 近期频繁报警
- 与高危人员有密切联系
- 活动轨迹异常

7.3 知识图谱关系构建规则
7.3.1 关系抽取规则
1. 从JQ_SMSJ_ontology表抽取市民-警情关系
2. 从JQ_MJFZ_ontology表抽取民警-警情关系
3. 通过同案分析构建市民-市民同行关系
4. 通过购买记录构建市民-物品关系
7.3.2 关系权重计算
关系权重 = 关联次数 × 时间衰减系数

7.4 定时任务执行规则
7.4.1 标签批量标注
- 执行时间：每日凌晨2:00
- 执行范围：最近24小时新增或更新的实体
