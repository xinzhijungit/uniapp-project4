## 1. 产品概述

### 1.1 功能定位

“警情预警”功能是“数据研判”模块的核心组成部分，旨在通过AI模型与自定义规则，对海量接警单数据进行实时监测与智能分析，主动发现高风险、异常波动的警情事件，并生成预警信息。产品将支撑指挥中心及各分局、派出所进行事前预防、事中处置与事后复盘的全流程闭环管理。

### 1.2 核心价值

*   **智能驱动**：基于AI模型与配置化指标，实现从被动接警到主动预警的转变。
*   **风险可视**：通过等级（红/橙/黄）和统计看板，直观展示区域治安态势。
*   **闭环处置**：提供从预警查看、警情关联到处置反馈的完整工作流。
*   **灵活配置**：允许业务专家通过配置指标和规则，快速适配不断变化的治安形势。

### 1.3 功能模块组成

1.  **预警管理**：预警信息的统一展示、多维筛选、详情查看与处置反馈。
2.  **预警指标**：定义预警的基础原子能力（如“同一地址短时警情激增指数”），作为预警规则的“原材料”。
3.  **预警规则**：组合一个或多个指标，设定触发条件、通知对象及预警等级，是生成预警事件的“加工厂”。

## 2. 用户角色与核心诉求

| 角色           | 核心诉求                                      |
| :----------- | :---------------------------------------- |
| **指挥中心民警**   | 宏观掌握全市警情态势，对高优先级（红色）预警进行跟踪，督办处置进展。        |
| **分局/派出所民警** | 接收本辖区内的预警通知，查看关联警情详情，反馈处置结果。              |
| **数据研判专家**   | 根据业务需要，创建、调整预警指标（如SQL指标）和预警规则，优化预警模型的精准度。 |
| **系统管理员**    | 进行基础的配置管理（如组织机构、用户权限）。                    |

## 3. 业务规则与数据说明

*   **数据来源**：所有预警相关分析均基于 `接警单表 (jjd_jjd)` 业务数据。
*   **预警事件生成**：系统后台（调度任务或实时流计算）定期校验 `预警规则`。当满足规则中的 `触发条件` 时，自动生成一条 `预警事件` 记录。
*   **预警事件状态流转**：
    *   **待处置**：新生成的预警，尚未被任何用户处理。
    *   **处置中**：用户已查看详情，或已填写部分处置内容但未最终提交。
    *   **已采纳**：用户填写了处置结果，并标记为“采纳”。
    *   **不采纳**：用户填写了处置结果，并标记为“不采纳”。
*   **预警等级**：
    *   **红色特急**：最高级别，需立即介入。
    *   **橙色紧急**：需重点关注，尽快处置。
    *   **黄色关注**：保持关注，常规处置。

## 5. 非功能性需求

### 5.1 性能要求

*   列表页（预警、指标、规则）首次加载及查询响应时间应在2秒以内。
*   预警详情页加载（含关联警情）应在1.5秒以内。
*   支持至少50个并发用户同时操作无明显延迟。

### 5.2 安全性

*   所有用户操作需基于现有的RBAC权限系统进行校验。
*   SQL指标必须经过严格的语法和危险操作校验，防止SQL注入。
*   用户间的数据需进行隔离，用户只能查看和操作其权限范围内的预警数据（如按辖区、部门）。

### 5.3 易用性

*   页面布局和交互遵循平台统一设计规范。
*   所有关键操作（删除、停用）均需提供二次确认弹窗。
*   表单提交失败时，需明确提示错误原因，并保留用户已输入的内容。
